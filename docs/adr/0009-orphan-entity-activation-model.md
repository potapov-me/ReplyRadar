# ADR-0009: Модель активации сущностей для фильтрации orphan entities

## Статус

Принято

## Контекст

EntityExtract запускается по всем сообщениям. В результате граф неизбежно накапливает сущности двух видов:

**Orphan entities** — люди и организации, упомянутые вскользь, без реального контекста в жизни пользователя. Примеры: публичные фигуры в пересланных новостях, «менеджер Сбера» из чужого рассказа, персонаж анекдота. У них нет связей с мониторируемыми чатами, нет фактов из первых уст, нет обязательств или вопросов без ответа.

**Релевантные сущности** — люди и организации, с которыми пользователь реально взаимодействует или которые системно влияют на его переписки.

Проблема не в уверенности извлечения (confidence это не решает), а в релевантности: сущность может быть извлечена с высокой уверенностью и при этом не иметь никакой ценности для пользователя. По мере накопления orphan entities деградируют поиск, транзитивные запросы и дайджест.

## Решение

Ввести поле `status` на таблице `entities` с тремя состояниями:

```
candidate  →  active
candidate  →  muted   (пользователь явно отключил)
active     →  muted
muted      →  active  (пользователь явно включил)
```

Все новые сущности создаются со статусом `candidate`. По умолчанию API, поиск и дайджест работают только с `active`. `candidate` хранятся и могут быть промоутированы — данные не теряются.

### Критерии автоматической активации

Сущность переходит в `active` автоматически при выполнении **любого** из условий:

1. **Прямое участие в чате** — `telegram_id` сущности совпадает с отправителем хотя бы одного сообщения в мониторируемом чате.
2. **Системная вовлечённость** — сущность фигурирует как автор или адресат в `commitments` или `pending_replies`.
3. **Повторяемость** — сущность упомянута в ≥ 3 разных сообщениях из ≥ 2 разных мониторируемых чатов.

Порог повторяемости (3/2) намеренно небольшой: цель не отсечь периферийных людей, а отсечь однократные упоминания без контекста.

### Ручное управление

- `POST /people/{id}/activate` — принудительная активация
- `POST /people/{id}/mute` — отключение (не появляется в поиске и дайджесте)
- `GET /people?status=candidate` — просмотр кандидатов

### Схема

Добавить в таблицу `entities`:

```sql
status              text NOT NULL DEFAULT 'candidate'  -- 'candidate' | 'active' | 'muted'
status_changed_at   timestamptz
mention_count       int NOT NULL DEFAULT 0             -- число уникальных сообщений-источников
activated_by        text                               -- 'auto' | 'user'
```

`mention_count` обновляется инкрементально при каждом новом источнике в `entity_fact_sources` или `entity_relation_sources`. Проверка критерия повторяемости происходит там же — после каждого инкремента.

## Последствия

- Граф не теряет данные: `candidate` сущности хранятся и могут быть активированы позже.
- Поиск и дайджест чистые по умолчанию без ручной работы пользователя.
- Критерии активации явные и отлаживаемые: видно, почему сущность активирована (`activated_by`, условие).
- Порог повторяемости (3/2) — начальная эвристика. Калибруется в `evals/` на реальных данных.
- Публичные фигуры из пересланных новостей отсекаются автоматически: они не участвуют в чатах и не достигают порога без намеренного форварда цепочки сообщений.
- `muted` не удаляет данные — пользователь может вернуть сущность в `active` без потери истории.

# ADR-0017: Импорт экспорта Telegram Desktop как резервный путь доступа к истории

## Статус

Принято

## Контекст

Для доступа к истории личных переписок через Telegram Client API (MTProto) необходимы `API_ID` и `API_HASH`. Эти credentials создаются на [my.telegram.org](https://my.telegram.org) через форму «API development tools».

В ходе разработки ReplyRadar выяснилось, что получить credentials штатным способом не удаётся. Telegram на стороне my.telegram.org отклоняет создание приложения при несоответствии между страной, к которой привязан номер телефона (Россия), и географическим положением IP-адреса. Telegram в России формально заблокирован, поэтому для доступа к сервисам Telegram используется нероссийский IP — и система расценивает такую комбинацию как подозрительную. В результате форма создания API application возвращает ошибку, credentials не выдаются.

Проблема не в коде ReplyRadar и не в правильности заполнения формы — это ограничение на стороне инфраструктуры Telegram. Попытки обойти его (смена IP, разные браузеры, повторные регистрации) результата не дали.

Без `API_ID` и `API_HASH` невозможны:
- realtime listener (Telethon)
- backfill через `iter_messages`

Принципиально это означает, что стандартный путь ingestion недоступен для определённой категории пользователей — тех, у кого российский номер и нероссийский IP одновременно.

### Рассмотренные альтернативы

| Вариант | Почему отклонён |
|---------|----------------|
| Использовать чужой `API_ID` (неофициальные клиенты) | Нарушение ToS Telegram; риск бана аккаунта |
| Использовать сторонний прокси для MTProto без credentials | Технически сложно, требует нестандартной библиотеки |
| Временный номер с «правильной» географией | Недоступно в данном контексте |
| Ждать разблокировки Telegram в России | Не поддаётся контролю; неизвестны сроки |
| Telegram Desktop Export (JSON) | Официально поддерживаемый механизм экспорта, не требует API credentials |

## Решение

Спроектировать `POST /import/telegram-export` — загрузку `result.json` из Telegram Desktop Export как самостоятельный третий режим ingestion, независимый от MTProto и API credentials.

Парсер (`ingestion/tg_export_parser.py`) нормализует формат экспорта к той же схеме, что используют listener и backfill. Вставка идёт через тот же `INSERT ... ON CONFLICT (chat_id, telegram_msg_id) DO NOTHING` — экспорт и MTProto-пути совместимы и могут сосуществовать без конфликтов.

При получении API credentials в будущем пользователь просто включает listener и backfill — данные, уже загруженные через import, останутся в БД и не задублируются.

## Последствия

- История чатов доступна без MTProto credentials — через официальный механизм экспорта Telegram Desktop.
- Realtime ingestion (новые сообщения) по-прежнему требует credentials; для пользователей без них система работает в режиме «периодический ручной импорт».
- Импорт не отражает редактирование сообщений после экспорта: в БД хранится текст на момент первого импорта (immutable raw messages by design).
- Функция имеет самостоятельную ценность за пределами обхода блокировки: для очень больших чатов, где backfill упирается в FloodWait, импорт предпочтительнее.
- Зависимость от конкретного формата `result.json` — если Telegram изменит схему экспорта, потребуется обновить парсер.

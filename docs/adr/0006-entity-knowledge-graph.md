# ADR-0006: База знаний о людях и организациях (Entity Knowledge Graph)

## Статус

Принято

## Контекст

В ходе переписок упоминаются люди и организации — их роли, связи, факты о них. Эта информация рассеяна по сообщениям и теряется. Для полноценной навигации по перепискам нужен персистентный слой, который аккумулирует знания об участниках: что человек говорил о себе, что о нём говорили другие, в каких сообщениях это зафиксировано.

Требования:
- охватывать всех упоминаемых людей и организации, не только участников чатов
- поддерживать транзитивные запросы ("через кого связаны X и Y")
- хранить провенанс каждого факта (источник, сообщение, дата)
- не смешиваться с дайджестом, но быть доступным как справочник

## Решение

Вводится подсистема **Entity Knowledge Graph** поверх существующего Postgres.

Три ключевых решения внутри:

**1. Единая таблица сущностей для людей и организаций.** Организации — полноправные узлы графа. Без этого транзитивные цепочки ("Иван директор фирмы → в фирме работает Елена → Елена встречается с Виктором") обрываются на уровне строковых значений.

**2. Разделение фактов и связей.** Факты (`entity_facts`) — атрибуты одной сущности. Связи (`entity_relations`) — рёбра между двумя сущностями. Транзитивные запросы реализуются через рекурсивные CTE по таблице связей.

**3. Отдельная стадия обработки EntityExtract.** Запускается батчами по всем сообщениям (не только `is_signal`), потому что упоминания людей встречаются в любом контексте. Один LLM-вызов на батч — дешевле, чем per-message. Идемпотентность через флаг `entities_extracted_at` на таблице `messages`.

## Последствия

- Добавляется новая стадия в processing pipeline: EntityExtract после Extract.
- Добавляется процедура entity resolution: при каждом извлечении LLM-сущность сопоставляется с существующими записями по embedding similarity, при неопределённости — LLM подтверждает merge.
- Backfill entity extraction запускается только для сообщений с `entities_extracted_at IS NULL` — консистентно с существующей моделью retry.
- База знаний не влияет на работу дайджеста напрямую: дайджест может ссылаться на сущности по имени и ID, но не зависит от их наличия.
- Ненаправленные связи индексируются в обе стороны на уровне запросов (не хранятся дважды).

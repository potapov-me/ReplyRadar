# ADR-0010: Точечное применение тактического DDD в knowledge-домене

## Статус

Принято

## Контекст

ReplyRadar не является классическим enterprise-приложением, где полный DDD оправдан. Большая часть системы — это pipeline обработки данных и CRUD поверх Postgres. Однако knowledge-домен (entities, facts, relations) содержит нетривиальные бизнес-правила, которые без явной организации начинают расползаться по стадиям обработки, роутам и репозиториям.

Конкретные симптомы, которые решает это решение:
- логика активации сущностей может оказаться частично в `entity_extract.py`, частично в `repos/entities.py`, частично в API-роуте
- правила superseding при противоречии фактов не имеют единственного места
- confidence calculation рискует дублироваться в summarizer и digest
- термины используются неоднородно: один и тот же concept называется по-разному в разных модулях

## Решение

Применять **тактический DDD точечно** — только там, где он снижает сложность, а не добавляет её.

### Что берём

**Ubiquitous language.** Термины домена зафиксированы в `docs/glossary.md`. Те же термины используются в именах классов, функций, полей БД и в комментариях. Переименование concept в одном месте требует переименования везде.

**Явное место для domain rules.** Модуль `knowledge/` содержит не только инфраструктуру графа, но и бизнес-правила knowledge-домена. Четыре области:

| Модуль | Правило |
|--------|---------|
| `knowledge/activation.py` | политика активации: критерии перехода candidate → active |
| `knowledge/confidence.py` | формула effective_confidence и пороги отображения |
| `knowledge/resolution.py` | правила merge при entity resolution, safe-unmerge |
| `knowledge/superseding.py` | семантика противоречия: когда факт supersedes другой, как обновляются счётчики |

**Защита границ контекстов.** Три правила, нарушение которых — архитектурный дрейф:
- `processing/` не импортирует из `usecases/` и `api/`
- `api/routes/` не импортирует из `db/repos/` напрямую — только через `usecases/`
- `knowledge/` не знает о `processing/` и `api/`

### Что не берём

- **Aggregates и Domain Events** — избыточны для single-user MVP с Postgres как источником истины
- **Repository pattern в DDD-смысле** — `db/repos/` это data access layer, не DDD repositories с identity tracking
- **Application Services как отдельный слой поверх usecases** — `usecases/` уже выполняет эту роль

## Последствия

- Новые domain rules пишутся в `knowledge/`, а не там, где они впервые понадобились.
- Глоссарий — живой документ: при введении нового термина он добавляется туда до написания кода.
- Нарушение границ контекстов видно на code review через импорты: запрещённый импорт — сигнал архитектурного дрейфа.
- Подход не требует фреймворков или новых зависимостей.

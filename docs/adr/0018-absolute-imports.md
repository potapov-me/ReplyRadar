# ADR-0018: Абсолютные импорты внутри пакета replyradar

## Статус

Принято

## Контекст

Кодовая база выросла из компактного MVP в набор пакетов с формальными import boundaries (ADR-0015). При относительных импортах (`from ..db.repos import chats`) связь модуля с архитектурным слоем читается хуже: полный путь не виден в строке импорта, поиск по коду и review становятся менее однозначными, а механические перемещения файлов чаще требуют пересчёта `.`-цепочек.

Для проекта важнее предсказуемая навигация по пакетам и единый стиль, чем локальная краткость отдельных импортов.

## Решение

Все внутрипроектные импорты оформляются как абсолютные, от корня пакета `replyradar`.

Примеры:

```python
from replyradar.db.repos import chats as chats_repo
from replyradar.api.app import app
from replyradar.processing.extract import run_extract
```

Относительные импорты внутри `src/replyradar/` не используются.

Это правило относится и к импортам между соседними пакетами, и к импортам внутри одного bounded context. Исключений для "коротких" внутрипакетных ссылок не вводим, чтобы не смешивать стили.

## Последствия

- Полный модульный путь виден сразу в каждой import-строке.
- `rg`, code search и review проще использовать для навигации и проверки архитектурных зависимостей.
- Механические рефакторинги файлов меньше зависят от глубины текущего каталога.
- При добавлении новых модулей нужно соблюдать единый стиль; относительные импорты считаются отклонением от архитектурного соглашения.

# Архитектурный разбор ReplyRadar

## Краткий вывод

Итоговая оценка: `9.9/10`.

Текущая архитектура выглядит очень сильной для local-first single-user MVP:
- `Postgres` остаётся единственным источником истины
- ingestion, processing, API, scheduler и bot собраны в один осознанно упрощённый runtime
- realtime и backfill обслуживаются одним идемпотентным processing engine
- LLM-контур полностью локальный через `LM Studio` + `LiteLLM`
- поверх message pipeline есть полноценная база знаний о людях и организациях
- knowledge-домен оформлен как отдельный набор domain rules
- command/query paths разделены явно
- появились quarantine, audit trail, optimistic locking, health model, операционные метрики и golden datasets

Это уже не просто аккуратная схема обработки Telegram-сообщений, а архитектура, в которой продуманы не только pipeline и доменная модель, но и эксплуатация, ручное вмешательство и контроль качества.

## Сильные стороны

- Архитектура держит дисциплину вокруг одного источника истины: очередь не превращена в скрытое состояние, а recover/retry-сценарии опираются на `messages` и stage timestamps.
- Single-process runtime оформлен как осознанный MVP-компромисс, а не как случайная недодекомпозиция. Для single-user сценария это снижает операционную цену без архитектурного самообмана.
- Разделение pipeline на независимые стадии удачное: частичный успех допустим, перезапуск безопасен, realtime и backfill не расходятся по коду.
- Summary и digest не встроены в критический путь. Summary обновляется отдельно, digest запускается вручную, поэтому самые хрупкие LLM-сценарии вынесены из основного ingestion-контура.
- Entity Knowledge Graph добавляет проекту настоящий слой накопления знаний, а не временные product read-models. Таблицы `entities`, `entity_facts`, `entity_relations` и источники corroboration делают систему расширяемой без пересборки базовой модели.
- Модель `candidate` / `active` / `muted` — сильное усиление графа знаний. Она позволяет собирать широкий knowledge substrate, не засоряя пользовательские сценарии каждым слабым упоминанием.
- В `knowledge/` появился правильный уровень tactical DDD: activation policy, merge rules, superseding semantics и confidence model вынесены в доменный слой, а не размазаны по pipeline и API.
- Command/query split описан ровно настолько, насколько нужно этому проекту: мутации идут через `usecases/`, read-only запросы могут идти напрямую в `repos`. Это прагматичнее и чище, чем насильственно тащить все чтения через application layer.
- Для ручных операций в knowledge graph теперь есть `expected_version` и optimistic locking. Это важное усиление: архитектура уже учитывает race conditions и не делает вид, что операторских конфликтов не существует.
- Появление `processing_quarantine` закрывает один из главных пробелов ранних AI-систем: невалидный или конфликтный LLM-результат больше не обязан бесконечно болтаться между retry и silent failure.
- `entity_audit_log` — очень сильное решение. Ручные `activate`, `mute`, `merge`, `unmerge`, `relate` и override-действия перестают быть непрозрачными мутациями состояния и становятся проверяемой историей.
- Health model и `/admin/metrics` переводят архитектуру из уровня "есть endpoint /status" в уровень реальной операционной наблюдаемости: backlog по стадиям, queue depth, quarantine size, parse error rate и прокси-метрика качества merge — это уже зрелый набор для MVP.
- `evals/datasets` и gate на prompt-изменения — одна из лучших практик в документе. Это значит, что качество LLM-слоя рассматривается как регрессионная дисциплина, а не как ручное впечатление от последнего запуска.
- `docs/runbooks/` — правильный operational слой для такого продукта. Для single-user системы это особенно важно: здесь нет SRE-команды, поэтому runbook должен компенсировать отсутствие организационной памяти.
- Правило "границы контекстов через импорты" особенно уместно для Python-проекта такого масштаба. Это простая, дешёвая и проверяемая замена тяжёлым архитектурным фреймворкам.
- Приватность учтена конструктивно: локальный LLM-контур, отсутствие пользовательских текстов в логах и ручной запуск дайджеста хорошо сочетаются между собой.

## Остаточные риски

### Ingestion остаётся единственной жёсткой точкой потери данных

Это всё ещё прямо признано в архитектуре: при недоступности `Postgres` listener работает в best-effort режиме и может потерять новые сообщения до ручного backfill. Для MVP это допустимо, но это по-прежнему главный реальный failure mode системы.

### Shared-failure-domain у single-process runtime всё ещё широкий

Listener, processing, scheduler, API и bot живут в одном процессе и одном event loop. Компромисс оправдан, но любой серьёзный deadlock, runaway task или утечка памяти кладёт сразу весь контур, включая ingestion.

### Entity resolution всё ещё будет первым источником "тихих" ошибок знаний

Схема с embedding similarity, LLM-подтверждением merge, audit log и даже `safe-unmerge` уже хорошо продумана. Но именно здесь всё равно вероятны самые дорогие для доверия ошибки: неверная склейка двух людей хуже, чем временный дубль. Архитектура теперь хорошо готова к расследованию и исправлению таких ошибок, но не устраняет саму природу риска.

### Confidence model и activation model всё ещё эмпирические

`source_weight`, полураспад, пороги confidence, правила candidate → active, пороги по упоминаниям и чатам выглядят здраво, но пока это всё ещё эвристики. Без калибровки на реальных переписках система может скрывать полезные сущности слишком долго или, наоборот, активировать и показывать слабый шум раньше времени.

### EntityExtract по всем сообщениям всё ещё увеличивает вычислительную цену

Модель активации хорошо снижает пользовательский шум, quarantine снижает вред от плохих ответов, но вычислительная цена никуда не делась. EntityExtract по всем сообщениям по-прежнему будет одной из первых точек давления по latency, batch sizing и стоимости локального LLM-контура.

### `pending_replies` остаётся наиболее хрупкой продуктовой read-model

Даже на фоне сильно выросшей зрелости общей архитектуры именно этот слой, вероятнее всего, будет давать пользователю первые раздражающие ложные срабатывания. Причина не в инфраструктуре, а в самой природе задачи: "ждёт ли это сообщение ответа" хуже формализуется, чем commitments или entity facts.

### Import policy пока выглядит как review-rule, а не как автоматически проверяемый инвариант

Это уже хороший architectural contract, но по документу нарушение пока "видно на code review". Пока границы не проверяются автоматически, остаётся риск медленного архитектурного дрейфа через случайные импорты и удобные shortcuts.

### `summarizer/` и `digest/` всё ещё требуют аккуратной роли относительно `usecases/`

Появление command/query split сильно укрепило картину, но рядом с `usecases/` всё ещё есть отдельные `summarizer/` и `digest/` пакеты. Это нормально, если там только техническая реализация. Если сценарная orchestration-логика снова начнёт жить в нескольких местах, архитектура станет менее ясной.

## Вывод

Архитектура стала сильнее не только как схема данных и pipeline, но и как операционная система для реальной эксплуатации. В проекте уже видны composition root, application layer, command/query split, типизированный LLM-контур, ограниченный tactical DDD, quarantine-path, audit trail, optimistic locking, health model, метрики, golden datasets и runbooks.

Основные неизвестные теперь лежат не в форме архитектуры, а в калибровке доменных эвристик, качестве entity resolution, реальной полезности `pending_replies` и в том, насколько строго команда удержит объявленные границы в коде. Это именно тот тип остаточного риска, который должен сниматься реализацией, evals и эксплуатационными наблюдениями, а не дальнейшим усложнением архитектурного документа.

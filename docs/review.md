# Архитектурный разбор ReplyRadar

## Краткий вывод

Итоговая оценка: `9.4/10`.

Текущая архитектура выглядит цельной и внутренне согласованной для local-first single-user MVP:
- `Postgres` остаётся единственным источником истины
- ingestion, processing, API, scheduler и bot собраны в один осознанно упрощённый runtime
- realtime и backfill обслуживаются одним идемпотентным processing engine
- LLM-контур полностью локальный через `LM Studio` + `LiteLLM`
- поверх message pipeline появилась отдельная база знаний о людях и организациях
- деградация описана явно, без притворной "надёжности по умолчанию"

Это уже не просто схема извлечения `commitments` и `pending_replies`, а компактная knowledge-oriented система с понятными границами, хорошим provenance и внятным путём повторной обработки.

## Сильные стороны

- Архитектура держит дисциплину вокруг одного источника истины: очередь не превращена в скрытое состояние, а все recover/retry-сценарии опираются на `messages` и stage timestamps.
- Single-process runtime оформлен как осознанный MVP-компромисс, а не как случайная недодекомпозиция. Для single-user сценария это снижает операционную цену без архитектурного самообмана.
- Разделение pipeline на независимые стадии удачное: частичный успех допустим, перезапуск безопасен, realtime и backfill не расходятся по коду.
- Summary и digest больше не встроены в критический путь. Summary обновляется отдельно, digest запускается вручную, поэтому самые хрупкие LLM-сценарии вынесены из основного ingestion-контура.
- Entity Knowledge Graph добавляет проекту настоящий слой накопления знаний, а не временные product read-models. Отдельные таблицы `entities`, `entity_facts`, `entity_relations` и источники corroboration делают систему расширяемой без пересборки базовой модели.
- Провенанс проработан лучше среднего для MVP: у фактов и связей есть источник, сообщение, временные метки, модель, версия prompt и механизм superseding вместо затирания истории.
- Confidence scoring для facts/relations выглядит зрелым решением: архитектура не делает вид, что все извлечённые утверждения одинаково надёжны, и заранее оставляет место для калибровки на реальных данных.
- Приватность учтена конструктивно: локальный LLM-контур, отсутствие пользовательских текстов в логах и ручной запуск дайджеста хорошо сочетаются между собой.

## Остаточные риски

### Ingestion остаётся единственной жёсткой точкой потери данных

Это прямо признано в архитектуре: при недоступности `Postgres` listener работает в best-effort режиме и может потерять новые сообщения до ручного backfill. Для MVP это допустимо, но это главный реальный failure mode системы, а не формальность в описании отказов.

### Shared-failure-domain у single-process runtime всё ещё широкий

Listener, processing, scheduler, API и bot живут в одном процессе и одном event loop. Компромисс оправдан, но любой серьёзный deadlock, runaway task или утечка памяти кладёт сразу весь контур, включая ingestion.

### Entity resolution будет первым источником "тихих" ошибок знаний

Схема с embedding similarity и LLM-подтверждением merge разумна, но именно здесь вероятны самые дорогие для доверия ошибки: неверная склейка двух людей хуже, чем временный дубль. Архитектура уже даёт ручной merge, но не описывает явного контура safe-unmerge или ревизии ошибочных объединений.

### Confidence model пока сильнее концептуально, чем эмпирически

`source_weight`, полураспад, штраф за противоречия и пороги `0.75 / 0.40` выглядят здраво, но пока это всё ещё хорошо сформулированные эвристики. Без калибровки на реальных переписках система может либо скрывать полезные факты, либо показывать спекулятивные связи слишком уверенно.

### EntityExtract по всем сообщениям увеличивает цену шума

Решение извлекать сущности не только из `is_signal` сообщений оправдано для полноты графа, но именно эта стадия, скорее всего, первой упрётся в стоимость, latency и накопление слабополезных сущностей. Это не ошибка архитектуры, а её ожидаемая точка напряжения.

### `pending_replies` остаётся наиболее хрупкой продуктовой read-model

Даже после усиления общей архитектуры именно этот слой, вероятнее всего, будет давать пользователю первые раздражающие ложные срабатывания. Причина уже не в отсутствии pipeline, а в самой природе задачи: "ждёт ли это сообщение ответа" хуже формализуется, чем commitments или entity facts.

### `docs/structure.md` пока недоописывает границы application layer

Файловая структура хорошо раскладывает код по техническим пакетам, но между `api/routes/`, `processing/`, `digest/`, `summarizer/` и `db/repos/` не показан явный слой use-case orchestration. Если роуты, scheduler и CLI начнут напрямую собирать сценарии из репозиториев и LLM-клиента, проект быстро придёт к "тонким папкам, толстым entrypoints". Для такой архитектуры полезен отдельный слой `services/` или `usecases/`, где живут сценарии уровня `run_backfill`, `summarize_chat`, `build_digest`, `merge_people`.

### Не хватает явного места для typed contracts и prompt assets

В структуре есть `prompt_version` в данных и отдельные LLM-стадии, но нет выделенного пакета для prompt templates, response schemas и версионируемых контрактов между стадиями. Без этого LLM-интеграция почти неизбежно расползётся по строковым литералам и ad-hoc JSON parsing внутри `classify.py`, `extract.py` и `entity_extract.py`.

### `main.py` выглядит кандидатом в будущий god-object

По описанию он одновременно собирает event loop, FastAPI, scheduler и запуск процессов. Для MVP это нормально, но именно здесь обычно начинает накапливаться lifecycle glue-код, который потом сложно тестировать и безопасно менять. Стоит держать `main.py` максимально тонким и выносить wiring в отдельный composition root-модуль.

### Терминология `api/deps.py` пока ближе к ORM-стилю, чем к заявленной модели

В `structure.md` указано `db session` в зависимостях API, хотя в проекте сознательно выбраны `asyncpg` и SQL-репозитории без ORM. Это мелочь, но такие несоответствия потом превращаются в архитектурный drift: код начинает выглядеть как псевдо-SQLAlchemy-приложение, хотя фактически им не является.

## Вывод

Архитектура стала сильнее по двум причинам: она честно удержала MVP-простоту и при этом добавила полноценный knowledge layer вместо локальных ad-hoc сущностей. Самое важное здесь не новые таблицы сами по себе, а то, что они встроены в уже существующую модель идемпотентной обработки, provenance и деградации.

Основные неизвестные теперь лежат не в структуре системы, а в качестве entity resolution, confidence calibration и реальной полезности `pending_replies` на живых данных. Это хороший тип риска: дальше проекту нужнее реализация, evals и эксплуатационные наблюдения, чем дальнейшее усложнение архитектурного документа.

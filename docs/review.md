# Архитектурный разбор ReplyRadar

## Краткий вывод

Итоговая оценка: `9.7/10`.

Текущая архитектура выглядит зрелой и внутренне согласованной для local-first single-user MVP:
- `Postgres` остаётся единственным источником истины
- ingestion, processing, API, scheduler и bot собраны в один осознанно упрощённый runtime
- realtime и backfill обслуживаются одним идемпотентным processing engine
- LLM-контур полностью локальный через `LM Studio` + `LiteLLM`
- поверх message pipeline появилась полноценная база знаний о людях и организациях
- добавлена модель активации сущностей, которая отделяет накопление знаний от пользовательской видимости
- файловая структура теперь отражает архитектуру, а не просто раскладывает код по техническим папкам

Это уже не просто схема извлечения `commitments` и `pending_replies`, а компактная knowledge-oriented система с понятными границами, хорошим provenance, управляемой деградацией и заметно более зрелым application layer, чем раньше.

## Сильные стороны

- Архитектура держит дисциплину вокруг одного источника истины: очередь не превращена в скрытое состояние, а recover/retry-сценарии опираются на `messages` и stage timestamps.
- Single-process runtime оформлен как осознанный MVP-компромисс, а не как случайная недодекомпозиция. Для single-user сценария это снижает операционную цену без архитектурного самообмана.
- Разделение pipeline на независимые стадии удачное: частичный успех допустим, перезапуск безопасен, realtime и backfill не расходятся по коду.
- Summary и digest не встроены в критический путь. Summary обновляется отдельно, digest запускается вручную, поэтому самые хрупкие LLM-сценарии вынесены из основного ingestion-контура.
- Entity Knowledge Graph добавляет проекту настоящий слой накопления знаний, а не временные product read-models. Таблицы `entities`, `entity_facts`, `entity_relations` и источники corroboration делают систему расширяемой без пересборки базовой модели.
- Модель `candidate` / `active` / `muted` — сильное усиление графа знаний. Она решает типичную проблему таких систем: можно собирать широкий knowledge substrate, не засоряя пользовательские сценарии каждым слабым упоминанием.
- Провенанс проработан лучше среднего для MVP: у фактов и связей есть источник, сообщение, временные метки, модель, версия prompt и механизм superseding вместо затирания истории.
- Confidence scoring для facts и relations выглядит зрелым решением: архитектура не делает вид, что все извлечённые утверждения одинаково надёжны, и заранее оставляет место для калибровки на реальных данных.
- Структура проекта заметно усилилась: `bootstrap.py` задаёт нормальный composition root, `usecases/` отделяет application layer от технических деталей, а `llm/prompts/` и `llm/contracts/` фиксируют контракт LLM-стадий вместо строковой анархии.
- `main.py` стал тонким entrypoint, а `api/routes/` по документу работают через `usecases/`, а не собирают сценарии напрямую из репозиториев. Это хороший признак, что архитектура начинает быть реализуемой без быстрого дрейфа.
- Приватность учтена конструктивно: локальный LLM-контур, отсутствие пользовательских текстов в логах и ручной запуск дайджеста хорошо сочетаются между собой.

## Остаточные риски

### Ingestion остаётся единственной жёсткой точкой потери данных

Это прямо признано в архитектуре: при недоступности `Postgres` listener работает в best-effort режиме и может потерять новые сообщения до ручного backfill. Для MVP это допустимо, но это главный реальный failure mode системы, а не формальность в описании отказов.

### Shared-failure-domain у single-process runtime всё ещё широкий

Listener, processing, scheduler, API и bot живут в одном процессе и одном event loop. Компромисс оправдан, но любой серьёзный deadlock, runaway task или утечка памяти кладёт сразу весь контур, включая ingestion.

### Entity resolution будет первым источником "тихих" ошибок знаний

Схема с embedding similarity и LLM-подтверждением merge разумна, но именно здесь вероятны самые дорогие для доверия ошибки: неверная склейка двух людей хуже, чем временный дубль. Архитектура уже даёт ручной merge, но не описывает явного контура safe-unmerge или ревизии ошибочных объединений.

### Confidence model пока сильнее концептуально, чем эмпирически

`source_weight`, полураспад, штраф за противоречия и пороги `0.75 / 0.40` выглядят здраво, но пока это всё ещё хорошо сформулированные эвристики. Без калибровки на реальных переписках система может либо скрывать полезные факты, либо показывать спекулятивные связи слишком уверенно.

### EntityExtract по всем сообщениям всё ещё увеличивает вычислительную цену

Новая модель активации хорошо снижает пользовательский шум: теперь не каждая извлечённая сущность попадает в API и digest. Но вычислительная цена никуда не делась. EntityExtract по всем сообщениям по-прежнему будет одной из первых точек давления по latency, batch sizing и стоимости локального LLM-контура.

### `pending_replies` остаётся наиболее хрупкой продуктовой read-model

Даже после усиления общей архитектуры именно этот слой, вероятнее всего, будет давать пользователю первые раздражающие ложные срабатывания. Причина уже не в отсутствии pipeline, а в самой природе задачи: "ждёт ли это сообщение ответа" хуже формализуется, чем commitments или entity facts.

### Activation model тоже потребует калибровки на живых данных

Статусы `candidate` / `active` / `muted` архитектурно очень уместны, но автоматические правила активации пока всё ещё эвристические: участие в мониторируемом чате, вовлечённость в signals, порог по упоминаниям и чатам. Здесь возможны два класса ошибок: полезные сущности останутся скрытыми слишком долго, а шумные будут активироваться слишком рано. Это уже не структурная проблема, а вопрос продуктовой настройки.

### Границу между `processing/` и `usecases/` придётся удерживать дисциплиной

Новая структура уже правильнее прежней, но теперь риск сместился: orchestration легко может расползтись между `processing/`, `usecases/`, `scheduler/`, `digest/` и API-роутами. Если pipeline-логика и пользовательские сценарии начнут смешиваться, формально хорошие папки не спасут от архитектурного дрейфа.

### `summarizer/` и `digest/` пока выглядят как параллельные центры сценарной логики

Появление `usecases/` решает большую часть проблемы, но рядом с ним всё ещё существуют `summarizer/` и `digest/` как самостоятельные пакеты. Это нормально, если они содержат только техническую реализацию. Но если значимая orchestration-логика останется одновременно и там, и в `usecases/`, код снова получит несколько конкурирующих application layers.

## Вывод

Архитектура стала сильнее по двум причинам: она честно удержала MVP-простоту и при этом добавила полноценный knowledge layer вместо локальных ad-hoc сущностей. Дополнительно усилилась и файловая структура: теперь в ней уже видны composition root, application layer и типизированный LLM-контур, а не только техническая декомпозиция по папкам.

Основные неизвестные теперь лежат не в базовой форме системы, а в качестве entity resolution, confidence calibration, настройке activation model, реальной полезности `pending_replies` и в том, насколько строго команда удержит границы между pipeline, use cases и read-model слоями. Это хороший тип риска: дальше проекту нужнее реализация, evals и эксплуатационные наблюдения, чем дальнейшее усложнение архитектурного документа.

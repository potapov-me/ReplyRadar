# Архитектурный разбор ReplyRadar

## Краткий вывод

Итоговая оценка: `9.3/10`.

Архитектура готова к старту разработки. Для single-user локального MVP она выглядит зрелой и прагматичной:
- `Postgres` как источник истины
- идемпотентная обработка по стадиям
- один processing engine для realtime и backfill
- API, спроектированный вокруг сценариев использования
- локальный LLM-контур
- явная модель деградации
- жизненный цикл данных и контур оценки качества в минимально достаточном виде

## Сильные стороны

- Архитектура честно называет свои компромиссы: `best-effort ingestion`, ручной дайджест, эмпирический RAG, прагматичная идентичность фактов.
- Повторная обработка описана достаточно подробно: backfill по диапазону, сброс стадий, мягкое выбытие фактов.
- Summary больше не хрупкий шаг на каждое сообщение, а управляемый пакетный компонент.
- Приватность учтена не декларативно, а через логирование без пользовательских текстов и жизненный цикл данных.

## Остаточные риски

### `source_fingerprint`

Это осознанный MVP-компромисс. Возможны коллизии, если одно сообщение содержит несколько похожих фактов.

### `pending_replies`

Они формализованы слабее, чем `commitments`, и скорее всего дадут первые ложные продуктовые срабатывания.

### RAG defaults

`threshold=0.75` и агрегация `top-3` пока эвристические. Это допустимо для MVP, но потребует реальной калибровки.

## Вывод

Существенных архитектурных пробелов больше нет. Следующий источник истины для проекта — уже реализация и наблюдения на реальных чатах, а не дальнейшее раздувание архитектурного документа.

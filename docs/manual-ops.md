# Ручные операции с knowledge graph

Операции над сущностями, которые требуют решения пользователя. Все они пишут в `entity_audit_log` и требуют `expected_version` для защиты от конфликтов.

---

## activate

**Что делает:** переводит сущность из `candidate` в `active`. После этого она появляется в поиске, дайджесте и транзитивных запросах.

**Когда использовать:** система не активировала сущность автоматически (не достигла порога упоминаний), но вы знаете, что этот человек или организация важны.

**Последствия:**
- Сущность сразу появляется в `GET /people` и поиске.
- `activated_by` = `'user'`, `status_changed_at` обновляется.
- Активация фиксируется в audit log.

**Отменить:** `mute` или подождать — автоматически активация не отменяется.

```
POST /people/{id}/activate
Body: {"expected_version": 3}
```

---

## mute

**Что делает:** переводит сущность в `muted`. Данные сохраняются, сущность исчезает из поиска и дайджеста.

**Когда использовать:** сущность была активирована (автоматически или вручную), но оказалась нерелевантной — публичная фигура из новостей, случайное упоминание.

**Последствия:**
- Исчезает из `GET /people`, поиска, дайджеста, транзитивных запросов.
- Факты и связи сохранены — можно вернуть через `activate`.
- Фиксируется в audit log.

**Отменить:** `activate`.

```
POST /people/{id}/mute
Body: {"expected_version": 3}
```

---

## merge

**Что делает:** объединяет две записи сущностей в одну. Все факты, связи и источники переносятся на `person_a`. Запись `person_b` помечается как поглощённая.

**Когда использовать:** система создала два отдельных профиля для одного человека (разные написания имени, Telegram ID не совпал при извлечении).

**Последствия:**
- `person_b` более не возвращается в поиске.
- Все `entity_facts` и `entity_relations` `person_b` переносятся на `person_a`.
- `person_a.aliases` пополняется именами из `person_b`.
- `person_a.mention_count` суммируется.
- Полный snapshot до и после — в audit log (`payload.before`, `payload.after`).
- Активация пересчитывается для объединённой записи.

**Это деструктивная операция.** Проверьте профили обоих перед merge.

**Отменить:** `unmerge` (требует ручного восстановления из audit log).

```
POST /people/{a}/merge/{b}
Body: {"expected_version_a": 5, "expected_version_b": 2}
```

---

## unmerge

**Что делает:** разделяет ранее склеенные сущности, восстанавливая состояние до merge.

**Когда использовать:** merge был ошибочным — два разных человека склеились в один профиль.

**Как работает:** система читает `payload.before` из соответствующей записи audit log и восстанавливает `person_b` как отдельную запись. Факты и связи, добавленные после merge, остаются на `person_a`.

**Ограничение:** восстанавливается состояние на момент merge. Факты, извлечённые после — не разделяются автоматически, требуют ручной проверки.

```
POST /people/{id}/unmerge
Body: {"audit_log_id": "uuid-of-the-merge-event"}
```

---

## relate

**Что делает:** вручную добавляет связь между двумя сущностями.

**Когда использовать:** система не извлекла связь автоматически, но вы знаете о ней из контекста за пределами мониторируемых чатов.

**Последствия:**
- Связь создаётся с `source_type = 'user'`, `base_confidence = 1.0`.
- Участвует в транзитивных запросах наравне с автоматически извлечёнными.
- Фиксируется в audit log.

```
POST /people/{a}/relate/{b}
Body: {
  "relation_type": "works_with",
  "claim_text": "добавлено вручную",
  "expected_version_a": 5
}
```

## Просмотр истории операций

```
GET /admin/entities/{id}/audit
```

Возвращает хронологический лог всех ручных и автоматических операций над сущностью. Каждая запись содержит `action`, `actor`, `version_from → version_to`, `payload.before/after`, `created_at`.

---

## Конфликты версий

Если между чтением и записью сущность была изменена другой операцией, API возвращает:

```json
{"error": "version_conflict", "current_version": 7, "expected_version": 5}
```

Перечитайте профиль (`GET /people/{id}`), убедитесь что изменение всё ещё нужно, повторите с актуальной версией.

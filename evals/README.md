# Evals

Офлайн-оценка качества LLM-стадий на фиксированных примерах.

Статус на текущий момент: в CLI реализованы только evals для `classify` и `extract`. Упоминания `entity_extract` ниже относятся к следующему этапу и пока не соответствуют реальной команде.

## Принцип

Evals — это регрессионная дисциплина, не разовая проверка. Любое изменение промпта, модели или порогов в `config/` требует прогона evals и сравнения с предыдущим baseline. Если метрики упали — изменение не мерджится.

Evals намеренно небольшие и ручные. Цель — поймать регрессию, а не измерить абсолютное качество.

## Когда прогонять

| Триггер | Обязательно |
|---------|-------------|
| Изменение промпта в `llm/prompts/` | да |
| Изменение Pydantic-схемы в `llm/contracts/` | да |
| Смена модели в `config/default.yaml` | да |
| Изменение порогов (confidence, activation) | нет, но рекомендуется |
| Обычный код без касания LLM-слоя | нет |

## Процесс

```bash
# Прогнать evals для стадии
uv run python -m replyradar eval classify
uv run python -m replyradar eval extract

# Зафиксировать текущий результат как новый baseline
uv run python -m replyradar eval classify --update-baseline
```

Если метрика упала ниже baseline — команда завершается с ненулевым кодом.

## Стадия Classify

### Задача

Определить, содержит ли сообщение семантически значимый контент (`is_signal`).

### Формат датасета

`datasets/classify/examples.jsonl` — один пример на строку:

```json
{"id": "cls_001", "text": "Ок, пришлю отчёт до пятницы", "sender": "Иван", "is_signal": true, "note": "явное обязательство"}
{"id": "cls_002", "text": "хаха точно", "sender": "Петя", "is_signal": false, "note": "реакция без содержания"}
{"id": "cls_003", "text": "кстати, ты не забыл про встречу завтра?", "sender": "Лена", "is_signal": true, "note": "вопрос без ответа, ожидает реакции"}
{"id": "cls_004", "text": "ладно", "sender": "Иван", "is_signal": false, "note": "подтверждение без нового содержания"}
{"id": "cls_005", "text": "я думаю это была странная ситуация с их стороны", "sender": "Маша", "is_signal": true, "note": "communication risk: эмоциональная оценка без ответа"}
```

### Правила разметки

**`is_signal = true` если сообщение содержит хотя бы одно из:**
- явное или подразумеваемое обязательство ("пришлю", "сделаю", "договорились")
- вопрос или просьбу, которые ожидают ответа адресата
- эмоциональную оценку или напряжение, которое остаётся без реакции
- важный контекстуальный факт (место встречи, дедлайн, решение)

**`is_signal = false` если:**
- реакция без содержания (хаха, ок, 👍, ладно)
- светская болтовня без информационной ценности
- пересказ общеизвестного без контекста для конкретного чата

**Спорные случаи:**
- Если сомневаетесь — ставьте `is_signal = true` и добавляйте `"ambiguous": true`. Лучше ложный сигнал, чем пропущенный.
- Сообщения от ботов и автоматических уведомлений — всегда `false`.

### Метрики

| Метрика | Порог для регрессии |
|---------|---------------------|
| Precision | не ниже baseline − 0.05 |
| Recall | не ниже baseline − 0.03 |
| F1 | не ниже baseline − 0.04 |

Recall важнее Precision: пропущенный commitment хуже лишнего сигнала.

## Стадия Extract

### Задача

Из signal-сообщения извлечь структурированные факты: commitments, pending_replies, communication_risks.

### Формат датасета

`datasets/extract/examples.jsonl`:

```json
{
  "id": "ext_001",
  "text": "Ок, я пришлю тебе отчёт до пятницы",
  "sender": "Иван",
  "recipients": ["Лена"],
  "expected": {
    "commitments": [
      {"author": "Иван", "target": "Лена", "content": "пришлёт отчёт", "due_hint": "пятница"}
    ],
    "pending_replies": [],
    "communication_risks": []
  }
}
```

```json
{
  "id": "ext_002",
  "text": "кстати, ты не забыл про встречу завтра?",
  "sender": "Лена",
  "recipients": ["Иван"],
  "expected": {
    "commitments": [],
    "pending_replies": [
      {"reason": "вопрос про встречу без ответа", "urgency": "high"}
    ],
    "communication_risks": []
  }
}
```

```json
{
  "id": "ext_003",
  "text": "я думаю это была странная ситуация с их стороны. не знаю что думать.",
  "sender": "Маша",
  "recipients": [],
  "expected": {
    "commitments": [],
    "pending_replies": [],
    "communication_risks": [
      {"type": "emotional_unresolved", "explanation": "эмоциональная оценка без реакции собеседника"}
    ]
  }
}
```

### Правила разметки — Commitments

**Включать:**
- явные обязательства: "пришлю", "сделаю", "отправлю", "договорились что я"
- подразумеваемые при согласии: "ок" в ответ на просьбу = commitment автора ок

**Не включать:**
- намерения без адресата: "хочу попробовать сделать" без контекста
- условные обязательства без ясного триггера: "если будет время, сделаю"

**Поля:**
- `author` — кто обязался (имя отправителя или явно упомянутое лицо)
- `target` — кому (получатель или null если неизвестен)
- `content` — суть в 2–5 словах, не цитата
- `due_hint` — дедлайн если упомянут явно, иначе null

### Правила разметки — Pending Replies

**Urgency:**
- `high` — прямой вопрос адресату, просьба требующая ответа, напоминание о договорённости
- `medium` — косвенный вопрос, мягкая просьба, информация ожидающая подтверждения
- `low` — риторический вопрос с контекстом, фоновый запрос без срочности

**Не включать:**
- вопросы без конкретного адресата ("интересно, кто это придумал")
- вопросы уже получившие ответ в том же контексте датасета

### Правила разметки — Communication Risks

**Типы:**
- `emotional_unresolved` — эмоциональное сообщение без реакции
- `long_silence_after_important` — важное сообщение без ответа долгое время (контекст нужен)
- `repeated_unanswered` — один и тот же вопрос/просьба повторяется

**Не включать как риск:**
- нейтральные паузы в переписке
- молчание после закрытых высказываний

### Метрики

Сравнение на уровне полей, не точных строк. Commitment считается совпавшим если совпадают `author` + `target` + суть `content` (fuzzy, 70% overlap).

| Метрика | Порог |
|---------|-------|
| Commitment precision | не ниже baseline − 0.05 |
| Commitment recall | не ниже baseline − 0.05 |
| Pending reply precision | не ниже baseline − 0.07 |
| Pending reply recall | не ниже baseline − 0.05 |
| Risk detection recall | не ниже baseline − 0.10 |

Для risks порог шире — это наиболее субъективная стадия.

## Добавление новых примеров

Примеры добавляются когда:
- система допустила ошибку на реальных данных (regression case)
- обнаружен новый класс спорных случаев
- добавляется поддержка нового типа факта

**Не добавлять:**
- синтетические примеры без реального прообраза
- очевидные случаи для "накрутки" метрик

Минимальный полезный датасет: **30–50 примеров на стадию**. При старте достаточно 15–20 чтобы зафиксировать первый baseline.
